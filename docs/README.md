##### プロジェクト概要

会社統合をきっかけに、独自で発展していたタレントマネジメントシステムをスクラッチで開発。入社した 2023 年度では、デリバリーのスピード感や品質の向上、レガシーなアーキテクチャなど技術的な課題感が存在していたため、上記の課題解消をするべくアーキテクチャ変更や DevOps の改善、各種 EOL 対応を中心に従事している。
また、テックリードとして、開発チームのマネジメントや技術的な課題解決をしている。

##### 課題

対応期限が迫っていた EOL 対応の積み残しや、デリバリーや品質の向上を図るためのアーキテクチャ変更など、技術的な課題が存在していた。
また、専任のインフラ担当者が不在であり、業務を推進できる体制が整っておらず、インフラの改善活動が進まない状況であった。

##### 取り組み/解決策

インフラ専任担当として、積み残してある業務の洗い出しから優先順位付けを行い、業務を進める体制を整えた。
また、関係者との密なコミュニケーションで、期間やリスクを共有し、スケジュールを調整することで EOL 対応やアーキテクチャ変更を進めた。
取り組んだ内容は、次の通り。

- Aurora や EC2、ミドルウェアの EOL 対応
- EC2 から ECS へのコンテナ移行
- インフラのコード化（Terraform）
- CI/CD 環境構築
- AWS アカウントのマルチアカウント化
- セキュリティ対策
- ログやメトリクスを Datadog で可視化
- コスト削減
- ドキュメントの整備

##### プロジェクト概要

オンラインプログラミングにおける様々な学習教材を提供するプラットフォームの toC 、toBtoC プロダクト開発。
事業拡大に合わせたインフラ・アプリのリアーキテクチャの必要性が高まっていた。

##### 課題

複数の Web アプリケーションが単一サーバー上で動作しており、誤ったデプロイや、肥大したメモリ使用率による OOM などによりサーバー停止し、全てのプロダクトへの影響がある状態となっていた。

##### 取り組み/解決策

事業としてクリティカルなプロダクトを優先して、ネットワークやサーバーを分離し、単一障害点を極力なくし、耐障害性を高めた。
そのために、現在どのような構成で動いているのかの理解を、インフラ・ミドルウェア・アプリケーションの各レイヤーの把握を徹底し、社内メンバーとのコミュニケーションを密に連携しながら推し進めた。
また、Terraform による IaC 化を行い、インフラの再現性を担保するようなアプローチを取った。

##### 成果

主要なコンテンツサービスを 3 つをネットワークとサーバーから分離し、耐障害性を高め、IaC により再現性のある構築を実施しつつ、構築時間の短縮を図った。

- バックエンドエンジニア 1 人

##### プロジェクト概要

LINE 内で完結する MaaS アプリを自社プロダクトとして新規に開発。
チームとして自社プロダクト開発は初の取り組みのため、ゼロベースでプロダクト開発の取り組みに挑戦。
技術的な課題だけでなく、スクラム開発手法の浸透や、技術開発メンバーと事業開発メンバーとのコミュニケーションなど組織的な課題も同時並行的に取り組む必要がある状況であった。

##### 担当

技術開発が 8 名程度、事業開発が 2 名程度のチーム。
主にバックエンド・インフラ領域を担当。

##### 課題

チーム内メンバーの多くはプロダクト開発やスクラム手法の経験がない中で、立ち上げフェーズの推進役を担えるメンバーがいなかった。また、メンバーそれぞれが何らかのプロジェクトにアサインされている状態だったため、業務時間の確保が困難で、プロダクトの立ち上げのハードルをさらに高くした。

##### 取り組み/解決策

スモールスタートでまずやってみて、全体へアウトプットし、それを叩き台として議論を促進させるやり方で、上記の課題へ取り組んだ。まずは一人の先輩を巻き込み、1 ヶ月以内にアウトプットを共有して、議論が進むことを目指し、計画を立てた。

技術的な観点から主に取り組んだことは以下となる。

- バックエンドベストプラクティスの策定
  - チーム開発においてコーディング規約や組織内でのベストプラクティスを一定定義した方がスムーズな開発を実施できるため、FastAPI を用いた API の先行実装に取り組んだ
- プロダクトとしてのインフラをコード化
  - 今後の展開も見据えた拡販でき、かつコスト最適なアーキテクチャを定義し、terraform から構築できるようコード化に取り組んだ

組織的な観点から主に取り組んだことは以下となる。

- スクラム開発の推進
  - スクラム開発導入の準備やスプリントイベントによるスクラムマスターの役割、また非エンジニアである事業開発メンバーへの手法説明などに取り組んだ
- たたき台作成を主体的に実施
  - 限られた時間の中、プロジェクトが多忙なメンバーの組織体制で進めていくにあたり、自主的にプロトタイプを作成することや、あらかじめ議論したいポイントを整理してファシリテートをすることで、効率的かつ生産的に推進

上記の取り組みを通して、組織としてプロダクト開発を進み始めることができ、スクラム開発におけるスプリントのサイクルが回り始めた。
こういったプロセスで開発したプロダクトをベースとして、実証実験に取り組むことも決まった。

##### 成果

エンジニアリング業務に留まらず組織推進の貢献を評価され、社内における MVP を受賞した。
